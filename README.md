# supabase-rust-gen

Generate type-safe Rust structs from your Supabase database schema. Like `supabase-js` type generation, but for the Rust ecosystem.

[![Crates.io](https://img.shields.io/crates/v/supabase-rust-gen.svg)](https://crates.io/crates/supabase-rust-gen)
[![Documentation](https://docs.rs/supabase-rust-gen/badge.svg)](https://docs.rs/supabase-rust-gen)
[![License: MIT OR Apache-2.0](https://img.shields.io/badge/License-MIT%20OR%20Apache--2.0-blue.svg)](LICENSE)

## Why?

Manually writing Rust structs for your Supabase tables is tedious and error-prone. Column names change, types drift, and nullable fields get missed. `supabase-rust-gen` eliminates this by:

- **Connecting directly** to your Supabase project's PostgREST endpoint
- **Reading the OpenAPI spec** to understand your exact schema
- **Generating idiomatic Rust** with proper Serde derives
- **Handling edge cases** like JSONB, arrays, nullable fields, and PostgreSQL types

## Features

-  **Zero-config for basic usage** — just set your env vars and run
- **`.env` file support** — automatically loads variables from `.env` via `dotenvy`
- **RLS-aware** — optionally generates security policy stub helpers
-  **Full CRUD query support** — generates `postgrest-rs` functions with `--with-queries` (select, insert, update, delete)
-  **Feature-flagged dependencies** — `chrono` and `uuid` enabled by default, opt out if needed
-  **Type-accurate mapping** — `int8` → `i64`, `jsonb` → `serde_json::Value`, etc.
-  **Self-documenting output** — includes column descriptions as doc comments
-  **Smart serialization** — nullable fields get `#[serde(skip_serializing_if = "Option::is_none")]`
-  **Async-first** — built on `tokio` and `reqwest`

## Installation

```bash
cargo install supabase-rust-gen
```

## Quick Start

```bash
# Set environment variables (or use a .env file)
export SUPABASE_URL="https://your-project.supabase.co"
export SUPABASE_ANON_KEY="eyJhbGciOiJIUzI1NiIs..."

# Generate types
supabase-rust-gen

# With all features enabled
supabase-rust-gen --with-rls --with-queries --output src/supabase_types.rs
```

Or create a `.env` file in your project root:

```env
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIs...
```

Then simply run:

```bash
supabase-rust-gen
```

## Usage

### CLI Options

```
supabase-rust-gen [OPTIONS]

Options:
  -u, --url <URL>          Supabase project URL [env: SUPABASE_URL]
  -k, --anon-key <KEY>     Supabase anonymous/public API key [env: SUPABASE_ANON_KEY]
  -o, --output <PATH>      Output file path [default: src/supabase_types.rs]
      --with-rls           Generate RLS stub helper functions
      --with-queries       Generate postgrest-rs CRUD query functions
  -h, --help               Print help
  -V, --version            Print version
```

### Generated Code Example

Given a Supabase table:

```sql
CREATE TABLE posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  content TEXT,
  published BOOLEAN DEFAULT false,
  metadata JSONB,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

Running `supabase-rust-gen` produces:

```rust
// Auto-generated Supabase types - DO NOT EDIT MANUALLY
//
// This file is generated by supabase-rust-gen.
// Run `supabase-rust-gen` to regenerate.

/// Row from the `posts` table
#[derive(Debug, Clone, serde::Deserialize, serde::Serialize)]
#[serde(rename_all = "snake_case")]
pub struct Posts {
    pub id: uuid::Uuid,
    pub title: String,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub content: Option<String>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub published: Option<bool>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub metadata: Option<serde_json::Value>,
    #[serde(skip_serializing_if = "Option::is_none")]
    pub created_at: Option<chrono::DateTime<chrono::Utc>>,
}
```

> **Note:** `uuid::Uuid` and `chrono::DateTime<chrono::Utc>` are used because the `chrono` and `uuid` features are enabled by default. Without these features, those fields would be `String`.

With `--with-queries` flag, you also get a full CRUD impl block:

```rust
impl Posts {
    /// Select all rows from the `posts` table
    pub async fn select_all_posts(
        client: &postgrest::Postgrest,
    ) -> Result<Vec<Self>, Box<dyn std::error::Error>> {
        let resp = client.from("posts").select("*").execute().await?;
        let text = resp.text().await?;
        let rows: Vec<Self> = serde_json::from_str(&text)?;
        Ok(rows)
    }

    /// Select rows matching a column filter
    pub async fn select_posts_eq(
        client: &postgrest::Postgrest,
        column: &str,
        value: &str,
    ) -> Result<Vec<Self>, Box<dyn std::error::Error>> {
        let resp = client.from("posts").select("*").eq(column, value).execute().await?;
        let text = resp.text().await?;
        let rows: Vec<Self> = serde_json::from_str(&text)?;
        Ok(rows)
    }

    /// Insert a single row
    pub async fn insert_one_posts(
        client: &postgrest::Postgrest,
        row: &Self,
    ) -> Result<Vec<Self>, Box<dyn std::error::Error>> { /* ... */ }

    /// Insert multiple rows
    pub async fn insert_many_posts(
        client: &postgrest::Postgrest,
        rows: &[Self],
    ) -> Result<Vec<Self>, Box<dyn std::error::Error>> { /* ... */ }

    /// Update rows matching a column filter
    pub async fn update_posts(
        client: &postgrest::Postgrest,
        column: &str,
        value: &str,
        updates: &serde_json::Value,
    ) -> Result<Vec<Self>, Box<dyn std::error::Error>> { /* ... */ }

    /// Delete rows matching a column filter
    pub async fn delete_posts(
        client: &postgrest::Postgrest,
        column: &str,
        value: &str,
    ) -> Result<Vec<Self>, Box<dyn std::error::Error>> { /* ... */ }
}
```

With `--with-rls` flag, you get RLS policy stub functions:

```rust
impl Posts {
    pub async fn can_select_posts(client: &Postgrest, user_id: Option<&str>) -> Result<bool, Box<dyn std::error::Error>> { /* ... */ }
    pub async fn can_insert_posts(client: &Postgrest, user_id: Option<&str>, row: &Self) -> Result<bool, Box<dyn std::error::Error>> { /* ... */ }
    pub async fn can_update_posts(client: &Postgrest, user_id: Option<&str>, row_id: &str) -> Result<bool, Box<dyn std::error::Error>> { /* ... */ }
    pub async fn can_delete_posts(client: &Postgrest, user_id: Option<&str>, row_id: &str) -> Result<bool, Box<dyn std::error::Error>> { /* ... */ }
}
```

> **Note:** RLS helpers are stubs — implement the actual policy checks based on your Supabase RLS configuration.

## Feature Flags

The CLI tool itself has compile-time feature flags that control what types are generated:

| Feature  | Default | Effect on Generated Code |
|----------|---------|--------------------------|
| `chrono` | ✅ Yes  | `timestamptz` → `chrono::DateTime<chrono::Utc>` instead of `String` |
| `uuid`   | ✅ Yes  | `uuid` → `uuid::Uuid` instead of `String` |
| `postgrest` | ❌ No | Required if using `--with-queries` or `--with-rls` at runtime |

To install without default features:

```bash
cargo install supabase-rust-gen --no-default-features
```

### Dependencies for Your Project

The generated code requires these dependencies in your project's `Cargo.toml`:

```toml
[dependencies]
# Always required
serde = { version = "1.0", features = ["derive"] }
serde_json = "1.0"

# Required if chrono feature was enabled (default)
chrono = { version = "0.4", features = ["serde"] }

# Required if uuid feature was enabled (default)
uuid = { version = "1.6", features = ["serde"] }

# Required if --with-queries or --with-rls was used
postgrest = "1.6"
```

## Type Mapping

| PostgreSQL Type | Rust Type (default) | Rust Type (with features) |
|----------------|---------------------|--------------------------|
| `text`, `varchar`, `char`, `bpchar` | `String` | |
| `int2`, `smallint` | `i16` | |
| `int4`, `integer` | `i32` | |
| `int8`, `bigint` | `i64` | |
| `float4`, `real` | `f32` | |
| `float8`, `double precision` | `f64` | |
| `numeric`, `decimal` | `f64` | Precision may be lost |
| `bool`, `boolean` | `bool` | |
| `json`, `jsonb` | `serde_json::Value` | |
| `bytea` | `Vec<u8>` | |
| `date`, `time`, `timetz` | `String` | |
| `timestamptz`, `timestamp` | `String` | `chrono::DateTime<chrono::Utc>` |
| `uuid` | `String` | `uuid::Uuid` |
| `ARRAY` | `Vec<T>` | Recursive mapping |
| Nullable column | `Option<T>` | + `skip_serializing_if` |
| Unknown types | `String` | Fallback |

## Real-World Example

```rust
use postgrest::Postgrest;

// Include the generated types
mod supabase_types;
use supabase_types::Posts;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let client = Postgrest::new("https://your-project.supabase.co/rest/v1")
        .insert_header("apikey", "your-anon-key");

    // Type-safe select all
    let posts = Posts::select_all_posts(&client).await?;
    
    for post in &posts {
        println!("{} - {}", post.id, post.title);
    }

    // Filtered select
    let published = Posts::select_posts_eq(&client, "published", "true").await?;
    println!("Published posts: {}", published.len());

    // Type-safe insertion
    let new_post = Posts {
        id: uuid::Uuid::new_v4(),
        title: "Hello Rust".to_string(),
        content: Some("Generated types are great!".to_string()),
        published: Some(true),
        metadata: None,
        created_at: None,
    };
    
    let inserted = Posts::insert_one_posts(&client, &new_post).await?;
    println!("Inserted {} post(s)", inserted.len());

    Ok(())
}
```

## How It Works

1. **Schema Fetching**: Calls `GET {SUPABASE_URL}/rest/v1/` with your API key to retrieve the OpenAPI specification
2. **Type Parsing**: Extracts table definitions, column types, nullability, and descriptions from the OpenAPI `definitions`
3. **Type Mapping**: Converts PostgreSQL types to Rust types based on compiled features (`chrono`, `uuid`)
4. **Code Generation**: Uses `quote` and `proc-macro2` to build valid Rust token streams, then formats with `prettyplease`
5. **Output**: Writes a single, self-contained Rust file ready for inclusion via `mod` or `include!`

## Project Structure

```
src/
├── main.rs              # CLI entry point (tokio async main)
├── lib.rs               # Public API: Cli, Generator, GeneratorConfig, SchemaFetcher
├── cli.rs               # Clap CLI argument definitions
├── output.rs            # File writing with parent directory creation
├── schema/
│   ├── mod.rs           # OpenAPI spec types (Definition, Property)
│   ├── fetch.rs         # HTTP fetcher for PostgREST OpenAPI schema
│   └── parse.rs         # OpenAPI → ParsedSchema (tables + columns)
├── types/
│   ├── mod.rs           # TypeMapper re-export
│   └── mapping.rs       # PostgreSQL → Rust type mapping logic
└── generate/
    ├── mod.rs           # Generator orchestrator (structs + rls + queries)
    ├── structs.rs       # Struct code generation with serde derives
    ├── queries.rs       # CRUD query function generation
    └── rls.rs           # RLS stub function generation
```

## Limitations & Tradeoffs

- **Numeric precision**: `numeric`/`decimal` maps to `f64`, which may lose precision for very large numbers
- **Custom types**: Enums and composite types currently map to `String` (support planned)
- **RLS introspection**: RLS policies require SQL access; current helpers are stubs that return `true`
- **Array dimensions**: Only 1D arrays are supported; multi-dimensional arrays fall back to best-effort mapping
- **Struct naming**: Table names are converted to PascalCase directly (e.g., `harvest_records` → `HarvestRecords`)
- **Schema scope**: Only the public schema is supported; `--schema` flag is not yet implemented
